## Приложение `pages`

### Назначение

Приложение `pages` отвечает за управление контентными страницами фронтенда.  
Оно позволяет в админке собрать страницу из упорядоченных блоков, а затем отдать её структуру по API в формате JSON, пригодном для фронтенда.

Ключевые задачи:
- **Страницы** (`Page`) с человекочитаемым названием и `slug`
- **Типы блоков** (`BlockType`) — справочник доступных блоков (hero, FAQ и т.п.)
- **Блоки страницы** (`PageBlock`) — какие блоки и в каком порядке расположены на странице
- **Элементы блока** (`BlockItem`) — заголовки, тексты, иконки внутри блока
- **API** для получения структуры страницы: `GET /api/pages/<slug>/`

---

### Модели

#### `Page`

Логическая страница фронтенда.

- **Поля**:
  - `slug` — уникальный идентификатор страницы (например, `home`, `about`)
  - `name` — название страницы для админки
  - плюс поля аудита от `TimeStampedModel`: `created_at`, `updated_at`
- **Связи**:
  - `blocks` — связанные блоки страницы (`PageBlock`, `related_name="blocks"`)
- **Назначение**: определяет сущность страницы и набор блоков, которые на ней отображаются.

#### `BlockType`

Справочник типов блоков, которые может отрисовать фронтенд.

- **Поля**:
  - `code` — уникальный код блока (например, `hero`, `applications`, `faq`)
  - `name` — название типа блока
  - `description` — необязательное текстовое описание
- **Назначение**: по `code` фронтенд выбирает конкретный UI‑компонент.

#### `PageBlock`

Конкретный блок на конкретной странице (без текстов, только тип и порядок).

- **Поля**:
  - `page` — ссылка на страницу (`Page`)
  - `block_type` — ссылка на тип блока (`BlockType`)
  - `order` — порядок блока на странице (чем меньше, тем выше)
- **Назначение**: описывает структуру страницы: какие блоки и в каком порядке размещены.

#### `BlockItem`

Элемент внутри блока: заголовки, тексты и иконки.

- **Поля**:
  - `block` — ссылка на `PageBlock`
  - `title` — заголовок (опционально)
  - `subtitle` — подзаголовок (опционально)
  - `content` — основной текст (опционально)
  - `icon` — ключ или путь до иконки, который использует фронтенд (опционально)
  - `order` — порядок внутри блока
  - плюс поля аудита от `TimeStampedModel`
- **Назначение**: наполнение блока конкретным контентом в заданном порядке.

### Админка

В `admin.py` зарегистрированы все модели:

- **`PageAdmin`**
  - **Список**: `name`, `slug`, `created_at`, `updated_at`
  - **Поиск**: по `name`, `slug`
  - **Сортировка**: по `slug`

- **`BlockTypeAdmin`**
  - **Список**: `code`, `name`
  - **Поиск**: по `code`, `name`
  - **Сортировка**: по `code`

- **`PageBlockAdmin`**
  - **Список**: `page`, `block_type`, `order`
  - **Фильтры**: по `page`, `block_type`
  - **Сортировка**: по `page`, `order`
  - **Inline**: `BlockItemInline` — позволяет редактировать элементы блока на той же странице.

- **`BlockItemAdmin`**
  - **Список**: `block`, `title`, `order`, `created_at`
  - **Поиск**: по `title`, `subtitle`, `content`
  - **Фильтры**: по `block__page`, `block__block_type`
  - **Сортировка**: по `block`, `order`

#### Типичный сценарий работы в админке

1. Создать несколько **типов блоков** (`BlockType`), например:
   - `code="hero"`, `name="Hero‑блок"`
   - `code="faq"`, `name="FAQ"`
2. Создать **страницу** (`Page`), например:
   - `slug="home"`, `name="Главная"`
3. Создать для этой страницы **блоки** (`PageBlock`) с нужными типами и порядком:
   - блок `hero` с `order=1`
   - блок `faq` с `order=2`
4. Добавить в каждый блок **элементы** (`BlockItem`) через inline:
   - для `hero`: заголовок, подзаголовок, текст и т.п.
   - для `faq`: вопросы‑ответы, каждый — отдельный элемент.

### API

- **URL**: `GET /api/pages/<slug>/`
- **Имя маршрута**: `pages:page-detail`
- **View**: `PageDetailAPIView` (`apps.pages.views`)
- **Назначение**: вернуть структуру страницы с блоками и элементами.

#### Авторизация

Используется пермишен `OnlyWithApiKeyOrFromFrontend` (`apps.core.permissions`):

- допускает запросы либо с корректным API‑ключом,
- либо с доверенного фронтенд‑источника (логика определена внутри пермишена).

#### Формат ответа

Сериализатор `PageSerializer` возвращает:

- **Для страницы**:
  - `slug`
  - `name`
  - `blocks` — список блоков
- **Для блока (`PageBlockSerializer`)**:
  - `id`
  - `block_type` — объект с полями `code`, `name`
  - `order`
  - `items` — список элементов блока
- **Для элемента (`BlockItemSerializer`)**:
  - `id`
  - `title`
  - `subtitle`
  - `content`
  - `icon`
  - `order`

Пример (упрощённо):

```json
{
  "slug": "home",
  "name": "Главная",
  "blocks": [
    {
      "id": 1,
      "block_type": { "code": "hero", "name": "Hero‑блок" },
      "order": 1,
      "items": [
        {
          "id": 10,
          "title": "PM Meetup",
          "subtitle": "Сообщество продакт‑менеджеров",
          "content": "Описание мероприятия...",
          "icon": "hero-main",
          "order": 1
        }
      ]
    }
  ]
}
```

---

### Использование на фронтенде

- Фронтенд вызывает `GET /api/pages/<slug>/` (например, `/api/pages/home/`).
- По полю **`block_type.code`** выбирается компонент (Hero, FAQ и т.п.).
- Контент блока берётся из массива **`items`** в порядке `order`.
- Благодаря этому структуру страницы можно менять в админке **без изменений кода фронтенда**.
